When creating an object/model, build it the simplest, cleanest way — at the origin, in a standard plane, with no target position baked in. Then **orient (rotate) and position it where it belongs using the `align...` family**, relative to other parts. Do not compute exact coordinates, and do not embed placement in a `Plane(origin=...)` or a `Pencil(start=...)`.

This separation applies to **factory methods** too: a method that creates an object returns it in its default pose and takes only parameters that affect the *object model* — never orientation/location parameters (a direction, a target side, a position). The caller orients (`rotate_*`) and locates (`align`) at the use site. E.g. `_dovetail(clearance=0)` builds the dovetail; the caller does `_dovetail().rotate_y(90 * out_dir).align(channel)...` — `out_dir` is orientation, so it lives at the call site, not in `_dovetail`.

**Why:** positioning by absolute coordinates is brittle, hard to read, and doesn't compose — it silently breaks when a dimension changes. Aligning relative to another solid's bounding box expresses *intent* ("sit on its right edge", "centre on it, shifted up 5") and survives parametric changes. It matters most for a part used in multiple places: build it once, place each instance with `align`.

Anti-pattern (avoid): `SmartBox(..., plane=Plane(origin=(joint_x, y_center, z_base + lock_t/2), x_dir=..., z_dir=...))` — target position is baked into the plane. Likewise `Pencil(plane, start=(target_x, target_y))` to place geometry. Build at the origin instead, then `.align(...)`.

**How to apply** (`src/sava/csg/build123d/common/`):
- `align_x / align_y / align_z(reference, alignment=Alignment.C, shift=0, plane=Plane.XY)` move self along one axis relative to `reference`'s bounding box. `reference=None` aligns to the **origin**. `align_axis` is the generic form; `align_xy`/`align_xz`/`align_yz`/`align_zxy`/`align_old` are multi-axis shorthands.
- `Alignment` (geometry.py) is a two-letter code `[reference edge][self direction]`, each L/C/R: e.g. `LR` = self.min at ref.min (self extends right from ref's left edge); `RR` = self.min at ref.max (self sits just past ref's right edge); `RL` = self.max at ref.max; `C` = centres aligned. `shift` nudges further (toward +axis).
- `align(reference, plane)` first **centres all three axes** on the reference, then returns an `AlignmentBuilder` for fluent per-axis overrides: `.x(Alignment, shift).y(...).z(...)`. A bare number is a centre-shift — `.z(15)` means `(C, 15)`. `.then()`/`.done()` returns the SmartSolid to keep chaining other ops. Because it re-centres unset axes, see [[reference_alignment_builder_defaults]].
- Orientation that's intrinsic to the part (e.g. which way a taper runs) is fine to set at build time, but keep it origin-anchored; use `rotate_*` to reorient instances and `align` to place them.

**Idioms in the repo:** `cap.align(channel).z(Alignment.RL, dim.wall_thickness)`; `part.align(other).x(Alignment.LL).y(Alignment.RL)`. Multi-part scenes assemble parts against each other this way — see [[reference_arrange_scene_pattern]] — rather than at absolute coordinates.

Companion to [[feedback_emit_idiomatic_style]] (build in default `Plane.XY`, single transform at the end): that covers clean local-frame *construction*; this covers *placement* — do it with `align`, not coordinates.

**Known exception — non-axis-aligned `SmartBox` that is then `fillet`ed:** a solid built in a rotated/custom plane (e.g. a `SmartBox` whose taper must run along a non-height axis, so `plane=Plane(x_dir=..., z_dir=...)` — note even `Plane.YZ` counts) carries a baked orientation. `move`/`align` then replay the SmartSolid orientation transform with **~1e-7 error in a coordinate** (measured: a face landed at 3.0000002 instead of 3.0). That sub-micron mismatch against an adjacent exact face makes a sliver face that breaks a following `fillet` **at every radius** — not just a marginal one. **Best fix: build AND fillet at the origin (small, exact coords), THEN `move` the finished solid into place.** The fillet must precede the move — a move after rounding is fine (the sub-micron error in final position is irrelevant to export), but a move before rounding poisons it. Magnitude matters: the same plate that fillets fine at the origin fails once `move`d to x≈30. This beats the earlier "build in place via `Plane(origin=...)`" workaround — origin-build-then-fillet-then-move keeps the clean separation and needs no custom plane. (Evidence is still mixed: one clean origin-OK/moved-FAIL pair, but rotated-then-moved solids have also filleted fine elsewhere — treat as the first suspect for a mysteriously failing fillet, not a hard rule, until more cases accumulate.) Placing an interior reference point (e.g. a bump *root* that isn't a bbox edge) is then a single `plate.move(x=joint, y=ref.y_mid + sign*offset)` derived from the reference's bbox — clean coordinate-relative placement, not literals.

**Companion insight — convex AND concave corners in one feature (genderless dovetail tab/socket):** to round a bump's *external* (tip) corners outward AND its *internal* (root, where it meets the slab/body) corners inward — as a dimensioned draft's 2D fillet does — **fuse the bump onto its slab first, then fillet the vertical edges.** OCCT rounds each edge by its own convexity automatically (convex→outward, concave→inward); a standalone box filleted before fusing rounds *all four* corners outward (wrong root). Do this fillet on the **small standalone slab+bump at the origin**, never on the full assembled channel (the channel's complex end face — floor+walls+rim meeting at the joint — makes the concave fillet fail outright). A concave fillet needs the flat slab to extend **≥ radius** beyond the bump root, so a too-narrow flange (e.g. 0.5 mm margin for R0.7) fails — widen it. For a *cut* feature (socket), give the cutter a sacrificial flange in empty space so its bump gets a concave root → the cut pocket's mouth rounds convex and interior concave (the complement of the mating tab). Seen on `cablechannel.py`'s dovetail connector (R0.7), `_create_lock_plate`.
